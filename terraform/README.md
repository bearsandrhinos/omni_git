# Terraform Configuration for Snowflake Semantic Views

This directory contains Terraform configuration for managing Snowflake semantic views generated from Omni topic YAML files.

## Files

- `provider.tf` - Snowflake provider configuration
- `versions.tf` - Terraform and provider version requirements
- `terraform.tfvars.example` - Example configuration file (copy to `terraform.tfvars`)
- `*.tf` - Generated semantic view resources (auto-generated, do not edit manually)

## Setup

1. **Copy the example configuration:**
   ```bash
   cp terraform.tfvars.example terraform.tfvars
   ```

2. **Edit `terraform.tfvars`** with your Snowflake credentials:
   - Get a Personal Access Token from Snowflake Web UI → User Profile → Personal Access Tokens
   - Fill in your account, user, warehouse, database, and role

3. **Initialize Terraform:**
   ```bash
   terraform init
   ```

4. **Review changes:**
   ```bash
   terraform plan
   ```

5. **Apply changes:**
   ```bash
   terraform apply
   ```

## Generated Resources

Semantic view resources are automatically generated by the `generate_semantic_views.py` script. Each topic file generates a corresponding `omni_<topic_name>_sv.tf` file.

**⚠️ Do not manually edit the generated `.tf` files** - they will be overwritten on the next generation. If you need customizations, modify the Omni YAML files instead.

## CI/CD

The GitHub Actions workflow automatically:
- Generates Terraform resources when Omni files change
- Runs `terraform plan` to preview changes
- Runs `terraform apply` when PRs are merged to main
- Commits generated files back to the repository

## Security

- `terraform.tfvars` is gitignored and should never be committed
- Use GitHub Secrets for CI/CD credentials
- Personal Access Tokens should be rotated regularly
